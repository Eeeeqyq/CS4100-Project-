# CS4100-Project-

# Ambient Music Recommendation Agent
### CS4100 Artificial Intelligence — Northeastern University

An AI system that recommends music based on your **physiological state and environment** — not just your listening history. A Hidden Markov Model infers your hidden psychological mood from wrist sensor data, and a Deep Q-Network learns which music to recommend to move you toward a better emotional state.

---

## The Problem

Spotify and Apple Music know what you liked last week. They don't know that you're stressed at 11pm, running at 7am, or stuck in an afternoon focus session. Every recommendation ignores the most relevant signal: **how you feel right now**.

## Our Solution

A two-stage AI pipeline:

1. **HMM (Hidden Markov Model)** — reads wrist sensor signals (activity intensity, activity type, time of day, weather) and infers your hidden psychological mood state across 6 categories
2. **DQN (Deep Q-Network)** — takes that mood belief state and learns which music mood bucket to recommend to improve your psychological state, trained using real pre/post emotional ratings as reward signal

The key insight: mood is *hidden* — you can't directly observe it from a heart rate sensor. The POMDP framework (HMM belief state → Q-learning) is the principled way to handle this.

---

## Architecture

```
Wrist Signals + Environment
  [HR intensity, activity type, time, weather]
              │
              ▼
    ┌─────────────────┐
    │   HMM (6 states) │  ← trained with Baum-Welch EM
    │  forward/viterbi │
    └────────┬────────┘
             │  belief state (6-dim probability vector)
             ▼
    ┌─────────────────┐
    │   DQN Agent     │  ← trained with Double DQN + experience replay
    │  8-dim state    │
    │  8 action buckets│
    └────────┬────────┘
             │  recommended music bucket
             ▼
    ┌─────────────────┐
    │  Music Library  │  ← SiTunes + PMEmo + Spotify Kaggle
    │  ~110K tracks   │
    └─────────────────┘
```

**Hidden States (HMM):**

| ID | State | Valence | Arousal |
|----|-------|---------|---------|
| 0 | stressed-high | low | high |
| 1 | anxious-neutral | low | medium |
| 2 | calm-focused | medium | medium |
| 3 | relaxed-low | high | low |
| 4 | happy-energetic | high | high |
| 5 | fatigued | low | low |

**Action Buckets (DQN):**

| Bucket | Label | Valence | Energy | Tempo |
|--------|-------|---------|--------|-------|
| 0 | dark-slow | low | low | slow |
| 1 | dark-fast | low | low | fast |
| 2 | intense-slow | low | high | slow |
| 3 | aggressive | low | high | fast |
| 4 | chill-study | med | low | slow |
| 5 | indie | med | low | fast |
| 6 | soulful | med | high | slow |
| 7 | energetic | med | high | fast |

## Datasets

| Dataset | Role | Size |
|---------|------|------|
| [SiTunes](https://github.com/JiayuLi-997/SiTunes_dataset) | Primary training + reward signal | 2,006 interactions, 30 users, 944 tracks |
| [PMEmo](https://github.com/HuiZhangDB/PMEmo) | Music emotion validation | 736 tracks with EDA-validated valence/arousal |
| [Spotify Kaggle](https://www.kaggle.com/datasets/maharshipandya/-spotify-tracks-dataset) | Extended music library | ~89,500 tracks |

SiTunes is the core dataset — it provides wrist physiological signals, environmental context, and crucially, **pre/post psychological ratings** (valence + arousal on a continuous scale) for each listening session. These ratings are used directly as the RL reward signal.

**Reward function:**
```
reward = 0.7 × Δvalence + 0.3 × Δarousal
+1 if score > 0.1  (emotional lift)
 0 if |score| ≤ 0.1  (no change)
-1 if score < -0.1  (emotional drop)
```

**Observation encoding (0–179):**
```
obs = wrist_obs × 9 + time_bucket × 3 + weather_bucket

wrist_obs (0–19) = intensity_bucket × 5 + activity_type
  intensity: 0=<10, 1=10-30, 2=30-80, 3=≥80
  activity:  0=still, 1=act→still, 2=walking, 3=lying, 4=running
time_bucket: 0=morning, 1=afternoon, 2=evening
weather:     0=sunny, 1=cloudy, 2=rainy
```

## Running the Project

> ⚠️ Always run all commands from the **project root** directory.

### Step 1 — Clean the data
Run the three notebooks in `notebooks/` in order:
```
notebooks/01_clean_situnes.ipynb
notebooks/02_clean_pmemo.ipynb      ← only if PMEmo downloaded
notebooks/03_clean_spotify.ipynb    ← only if Spotify downloaded
```
Outputs land in `data/processed/`.

### Step 2 — Train the HMM
```bash
python src/hmm/hmm_train.py
```
- Trains a 6-state discrete HMM using Baum-Welch EM
- 3 random restarts, keeps best log-likelihood
- Runtime: ~5 minutes on CPU
- Output: `models/hmm.npz`

### Step 3 — Train the RL agent
```bash
python train_agent.py
```
- Requires `models/hmm.npz` from Step 2
- Trains Double DQN with experience replay for 2,000 episodes
- Runtime: ~10 minutes on CPU
- Output: `models/agent.pt`, `models/training_log.csv`

### Step 4 — Run the demo
```bash
python demo.py
```
Simulates 3 sessions (stressed student, morning run, afternoon study) showing the full pipeline in action.

## Key Design Decisions

**Why HMM instead of direct classification?**
Mood is a *hidden* variable — you can't directly observe it from wrist signals. A person sitting still at 11pm with elevated HR could be stressed or simply relaxed. The HMM models mood as a latent state that generates the observable signals, which is the mathematically correct framing.

**Why Double DQN?**
Standard DQN overestimates Q-values by using the same network to select and evaluate actions. Double DQN uses the online network to select the best action but the target network to evaluate it, significantly reducing overestimation bias and improving stability on small datasets like SiTunes.

**Why reward = 0.7×Δvalence + 0.3×Δarousal?**
Valence (happy/sad) is more reliably shifted by music than arousal (energy level). The weights sum to 1.0 and were chosen based on music psychology literature showing valence as the primary axis of music-induced emotion change.

**Why collapse high-valence into medium for action buckets?**
With only 8 buckets, splitting valence into 3 levels (low/med/high) produces 12 buckets, with some having very few tracks. Collapsing high into medium keeps all 8 buckets well-populated and consistent across all three music datasets.
